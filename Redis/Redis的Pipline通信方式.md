Redis Pipeline（管道）的通信模式本质上是**半双工**的，而非全双工。

### 关键原因：TCP 通信的 “请求 - 响应” 批次化

Pipeline 的核心是**将多个命令打包成一个批次发送**，减少 TCP 往返次数（RTT），但它仍遵循 TCP 协议的 “请求 - 响应” 模式，具体表现为：

1. **客户端先批量发送所有命令**：客户端一次性将多个命令写入 TCP 缓冲区并发送，此时客户端处于 “发送状态”，不会同时接收响应（半双工的 “单向发送” 阶段）。
2. **服务端批量返回所有响应**：服务端接收完所有命令后，按顺序执行并将结果批量返回给客户端，此时服务端处于 “发送状态”，客户端处于 “接收状态”（半双工的 “单向接收” 阶段）。

整个过程中，客户端和服务端**不会同时进行双向数据传输**（全双工的核心特征），而是 “先客户端发送所有请求，再服务端返回所有响应” 的单向交替模式，因此属于半双工。

### 与全双工的对比

- **全双工**：通信双方可以同时发送和接收数据（如 HTTP/2 的多路复用，客户端可在接收响应的同时继续发送新请求）。
- **Redis Pipeline**：客户端必须等所有命令发送完毕后，才能开始接收响应；服务端也必须等所有命令接收完毕后，才能开始发送响应，无法同时双向传输，因此不符合全双工定义。

### 总结

Redis Pipeline 通过 “批量请求 + 批量响应” 优化了性能，但通信模式仍基于 TCP 的半双工 “请求 - 响应” 模型，而非全双工。其核心价值是减少 TCP 交互次数，而非改变通信方向的并行性。
