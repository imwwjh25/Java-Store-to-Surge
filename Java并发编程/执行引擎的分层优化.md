### JVM 执行引擎（Execution Engine）

JVM 执行引擎主要由 **解释器（Interpreter）** 和 **即时编译器（JIT Compiler）** 组成，例如：

- C1 Client Compiler
- C2 Server Compiler
- Graal JIT

#### 1️⃣ 解释器（Interpreter）

解释器负责逐条解释执行字节码指令，其特点是：

- 启动速度快
- 无需预编译
- 适合程序初期执行
- 但长期运行性能较低

由于不需要等待编译完成，因此可以快速响应，但在高频执行场景下效率不高。

------

#### 2️⃣ 即时编译器（JIT Compiler）

JIT 编译器会将“热点代码”（Hot Spot Code）编译为本地机器码，并缓存到 **Code Cache** 中。

热点代码通常基于以下指标判定：

- 方法调用次数（Invocation Counter）
- 循环回边次数（Back Edge Counter）

编译后的机器码可以直接运行，从而显著提高长期执行性能。

------

#### 3️⃣ 分层编译（Tiered Compilation）

现代 JVM 采用 **分层编译策略（Tiered Compilation）**，综合解释器与不同级别 JIT 编译器的优点：

| 层级      | 执行方式 | 说明                       |
| --------- | -------- | -------------------------- |
| Level 0   | 解释执行 | 启动阶段，快速响应         |
| Level 1–2 | C1 编译  | 带基础性能监控与轻量优化   |
| Level 3–4 | C2 编译  | 深度优化，生成高性能机器码 |

执行流程通常为：

解释执行 → C1 编译 → C2 编译

这种策略实现了：

- 快速启动
- 渐进优化
- 长期高性能运行

------

#### 4️⃣ Code Cache

- 用于存储 JIT 编译后的机器码
- 本质上是 JVM 内存区域的一部分
- 受参数 `-XX:ReservedCodeCacheSize` 限制

如果 Code Cache 空间耗尽：

- JVM 可能停止进一步 JIT 编译
- 严重时可能退化为纯解释执行
- 导致性能明显下降

因此在高负载或长时间运行的服务中，需要合理配置 Code Cache 大小。

------

### 一句话总结

JVM 执行引擎通过“解释执行 + 分层 JIT 编译”的方式，实现：

> 快速启动 + 动态优化 + 长期高性能运行
