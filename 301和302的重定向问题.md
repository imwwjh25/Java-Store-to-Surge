301 永久重定向的返回地址（目标 URL），存储在 HTTP 响应头的 **`Location` 字段** 中 —— 这是 HTTP 协议的标准规定，所有遵循 HTTP 规范的服务器（如 Nginx、Tomcat）和浏览器，都会通过该字段传递和解析重定向目标。

## 一、核心结论：返回地址在 `Location` 响应头字段



当服务器返回 301 状态码时，必须在响应头中携带 `Location` 字段，其值就是最终的 “永久重定向目标地址”。

- 格式：`Location: 目标URL`（URL 可以是绝对地址，也可以是相对地址）；
- 浏览器行为：收到 301 响应后，会自动读取 `Location` 字段的值，跳转到该地址，且会 **缓存这个重定向关系**（后续访问原地址时，直接从本地缓存读取目标地址，无需再次请求服务器）。

## 二、实际示例（直观理解）



以 “原地址 `http://old.jd.com` 永久重定向到 `https://new.jd.com`” 为例，服务器返回的 301 响应如下（可通过浏览器 F12 开发者工具的「Network」面板查看）：

### 响应状态行：



http

```
HTTP/1.1 301 Moved Permanently
```



### 核心响应头（含 `Location`）：



http

```
Server: nginx
Date: Tue, 04 Nov 2025 10:00:00 GMT
Location: https://new.jd.com  # 301 重定向的目标地址，核心字段
Cache-Control: public, max-age=31536000  # 浏览器缓存重定向的时长（1年）
Content-Length: 0
```



### 关键说明：



1. `Location` 字段的值是绝对地址（`https://new.jd.com`），也支持相对地址（如原地址是 `http://old.jd.com/page`，`Location: /new-page` 表示重定向到 `http://old.jd.com/new-page`）；
2. `Cache-Control: max-age=31536000` 表示浏览器会缓存该 301 重定向关系 1 年 —— 期间再次访问 `http://old.jd.com` 时，浏览器不会向服务器发送请求，直接跳转到 `https://new.jd.com`（这是 “永久重定向” 的核心特性）。

## 三、补充：301 重定向的缓存机制（与 `Location` 相关）



301 的 “永久” 特性，本质是 **浏览器对 `Location` 字段的缓存**，缓存相关的响应头会辅助控制缓存行为：

- `Cache-Control`：指定缓存时长（如 `max-age=31536000` 表示 1 年）；
- `Expires`：指定缓存过期时间（如 `Expires: Wed, 05 Nov 2026 10:00:00 GMT`），优先级低于 `Cache-Control`；
- 若未指定缓存头，浏览器会使用 “默认缓存策略”（不同浏览器可能不同，通常会缓存较长时间）。

### 清除缓存的场景：



若后续需要修改 301 重定向的目标地址，需：

1. 服务器端修改 `Location` 字段的值；
2. 同时修改缓存头（如 `max-age=0` 或 `Cache-Control: no-cache`），强制浏览器重新请求服务器，获取新的 `Location` 地址。

## 四、与 302 重定向的区别（避免混淆）



302（临时重定向）的返回地址同样存储在 `Location` 字段中，但核心差异在 **缓存行为**：

- 301：浏览器缓存 `Location` 地址，后续直接跳转（永久生效）；
- 302：浏览器不缓存 `Location` 地址，每次访问原地址都会向服务器请求，获取最新的 `Location`（临时生效）。

## 五、总结



1. 301 永久重定向的返回地址，唯一存储在 HTTP 响应头的 **`Location` 字段** 中；
2. 浏览器通过解析 `Location` 字段实现自动跳转，并缓存该地址（永久重定向的核心）；
3. 查看方式：浏览器 F12 → Network → 选中 301 状态的请求 → 查看「Response Headers」中的 `Location` 字段。

简单说：`Location` 字段就是 301 重定向的 “目标地址指针”，是浏览器实现跳转的核心依据。
